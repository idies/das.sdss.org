SDSS Data Model

The SDSS Data Model is THE definition of the retained data in the survey, as well as all interface data between independent components of the survey (e.g., between separate pipelines, or between pipelines and the Operational Database). All pipelines, as well as the Operational Databases, must be in compliance with the Data Model.

The SDSS Data Model uses the formal methodology of Rumbaugh's Object Modeling Technique (Rumbaugh, J., Blaha, M., Premerlani, W., Eddy, F., and Lorensen, W. 1991, Object-Oriented Modeling and Design, Prentice Hall, Englewood Cliffs, N.J.), as implemented by the object-oriented CASE tool Paradigm Plus (hereafter PP). The official survey definition of all data is stored within Paradigm Plus. Paradigm Plus provides numerous views on this Data Model, as well as the ability to automatically generate C++ code, FITS file formats, and other interfaces to the survey retained data derived directly from the Data Model.

The Data Model is essentially complete for the Monitor Telescope, the DA for Imaging Array, the main imaging pipelines (ssc, astrom, ps, and frames), and target selection. The spectroscopic aspects of the survey are yet to be incorporated, but are currently under development.

This set of documentation describes our use of a formal Data Model to define the survey retained data; how to use Paradigm Plus to interact with the Data Model; and presents various views of the Data Model.

Flat-File Formats

The primary view of the Data Model for the user are the flat-file formats. These DEFINE the formats for ALL inputs and outputs for all pipelines. All inputs are in the form of flat-files, which are found in two flavors: FITS files (both images and binary tables), and FTCL parameter files (i.e., files readable with the DERVISH utility param2Chain. A complete list of all flat-file formats for the set of pipelines and other survey components currently supported by the Data Model may be found here.

TCL procs are available to check the format of DA/pipeline output flatfiles against the data model. Their use is described here.
